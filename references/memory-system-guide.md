# Memory 시스템 가이드

전문가 에이전트의 지속적 학습을 위한 Memory 시스템 설계 가이드입니다.

## 디렉토리 구조

```
memory/
├── MEMORY.md               # 엔트리포인트 (200줄 이내)
├── service-context.md       # 컨설팅 대상 서비스 맥락
├── task-log.md              # 작업 이력
├── lessons-learned.md       # 학습 기록
└── user-preferences.md      # 사용자 선호도
```

## 각 파일의 역할

### MEMORY.md (엔트리포인트)
- **200줄 이내** 유지 (초과 시 자동 로딩 범위를 벗어남)
- 최신 핵심 학습 사항을 상위에 배치
- 중요도 순으로 정렬, 오래된 내용은 하위 파일로 이동
- 포함 내용: 핵심 학습 요약, 자주 하는 실수와 해결법, 사용자 핵심 선호도, 다른 Memory 파일 참조 링크

### Context 파일

context 파일은 컨설팅 대상의 핵심 맥락을 담는 파일입니다.
모든 작업 시작 전 필수로 읽으며, 작업 중 새롭게 확인된 사실은 write-back합니다.

**공통 규칙:**
- **READ**: 모든 작업 시작 전 필수 참조 — skill 실행 시에는 워크플로우 0단계에서 자동으로 포함
- **WRITE-BACK 트리거 2가지**: (1) 사용자가 관련 새 정보를 직접 제공한 경우 (2) 작업 중 해당 파일 범위의 사실이 새롭게 확인/검증된 경우
- **기록 범위**: 확인된 사실만 기록 — 작업별 산출물(카피, 분석 결과 등)은 기록하지 않음
- MEMORY.md와의 관계: MEMORY.md는 전체 프로젝트 현황 요약, context 파일들은 각 영역의 상세 맥락
- **새 context 파일 추가 시**: 이 목록에 추가하면 됨 (CLAUDE.md, skill 수정 불필요)

**현재 context 파일 목록:**

| 파일 | 범위 | 내용 |
|------|------|------|
| `service-context.md` | 컨설팅 대상 서비스 | 회사 정보, 고객 문제 가설, 솔루션 가설, 현재 단계, 핵심 포인트, 리스크 |

### task-log.md (작업 이력)
- 수행한 작업의 구조화된 기록, **최근 순 정렬** (최신이 상단)
- 형식:
  ```markdown
  ## [날짜] 작업명

  - **사용자 요청**: 사용자가 요청한 내용
  - **처리 방안**: 어떻게 처리했는지 (도구, 단계, 방법)
  - **결과**: 산출물, 변경된 파일 등
  - **특이점**: 예상치 못한 상황, 제한사항 (없으면 "없음")
  - **배운점**: 이 작업에서 얻은 교훈
  ```
- 반복 패턴 발견 시 MEMORY.md에 요약

### lessons-learned.md (학습 기록)
- 실수와 조치 내역
- 발견한 패턴과 인사이트
- 형식: `[날짜] 상황 → 문제 → 해결 → 교훈`

### user-preferences.md (사용자 선호도)
- 선호하는 톤, 형식, 깊이
- 자주 요청하는 작업 패턴
- 싫어하는 것, 피해야 할 것

## Memory 관리 원칙

1. **최신 우선**: 가장 최근 학습이 MEMORY.md 상단에 위치
2. **중복 제거**: 같은 교훈이 반복되면 하나로 통합하고 빈도 표시
3. **주기적 정리**: 50개 이상 항목 누적 시 오래된 것을 요약/아카이브
4. **실행 가능한 형태**: "~했다"가 아닌 "~해야 한다" 형태로 기록
5. **크기 제한**: MEMORY.md 200줄 이내, 개별 파일 500줄 이내

## 세션 핸드오프

세션 간 작업 연속성을 보장하기 위한 규칙입니다.

### MEMORY.md `## 다음 세션 할 일` 섹션
- **위치**: `## 프로젝트 현황` 바로 아래에 배치
- **용도**: 세션 재시작/종료 전 미완료 작업을 기록
- **형식**: 작업 내용 + 배경 맥락 (새 세션에서 맥락 없이도 이해할 수 있도록)
- **관리**: 새 세션에서 해당 작업 완료 후 섹션 내용을 비움 (섹션 자체는 유지)

### 세션 핸드오프가 필요한 상황
- MCP 서버 추가 등 세션 재시작이 필요한 설정 변경
- 사용자가 세션 종료를 요청했으나 진행 중인 작업이 있는 경우
- 컨텍스트 한계로 세션 전환이 필요한 경우

### 기록 절차
1. 미완료 작업을 MEMORY.md `## 다음 세션 할 일`에 기록
2. 작업 배경과 다음 단계를 구체적으로 명시 (예: "youtube-data MCP 서버 추가 완료 → 9개 Tool 테스트 필요")
3. 사용자에게 "다음 세션에서 이어서 진행하겠습니다"라고 안내

### 새 세션 시작 시
1. MEMORY.md 읽기 → `## 다음 세션 할 일` 확인
2. 미완료 작업이 있으면 사용자에게 즉시 안내: "이전 세션에서 {작업}이 남아있습니다. 이어서 진행할까요?"
3. 작업 완료 후 해당 섹션 내용 비움

## Memory vs Knowledge vs User Inputs 구분

Memory, Knowledge, User Inputs는 보완 관계이며, 목적과 관리 방식이 다릅니다.

| 구분 | Memory | Knowledge | User Inputs |
|------|--------|-----------|-------------|
| 정체성 | 에이전트의 경험 기록 | 에이전트가 수집한 도메인 지식 | 사용자가 제공한 원본 자료 |
| 작성 주체 | 에이전트 | 에이전트 | 사용자 |
| 가공 여부 | 에이전트가 정리 | 에이전트가 정리 | 원본 그대로 |
| 참조 트리거 | 세션 시작, skill 실행 전 | 모든 작업 시작 전 | 모든 작업 시작 전 (index.md) |
| 관리 방식 | 오래된 것 탈락 | 오래된 것 재검증 | 사용자 요청 시에만 삭제 |
| 크기 제한 | MEMORY.md 200줄 | 없음 | 없음 |
| 기록 시점 | 작업 완료 후 | 대화 중 즉시 (웹 서칭 직후) | 사용자 자료 제공 시 |
